## 5. 跑 800 条 caption 正式实验

这一部分对应：

* `public_subset_extended.tsv`
* 或 `public_subset_core.tsv`

推荐先跑 `extended`，再跑 `core`。

---

## 5.1 运行批量推理脚本

执行：

```bash id="o5dlyr"
bash /root/experiment_suite/run_full_800.sh
```

---

### 5.1.1 这一步做了什么

该脚本会：

1. 激活 `vlmeval` 环境
2. 进入 `VLMEvalKit`
3. 调用：

```bash id="q9fhoz"
python run.py --data public_subset_extended --model GPT4o --mode infer --api-nproc 1 --work-dir /root/autodl-tmp/outputs
```

也就是说，它会对 **800 条实验集** 批量调用 GPT-4o 生成图像描述。

---

### 5.1.2 运行时间预期

由于设置为：

```text id="5ss9e2"
--api-nproc 1
```

表示单并发 API 调用，所以整体时间会比较长。
如果你前面 50 条大约花了 8 分钟，那么 800 条可能需要数小时。

---

### 5.1.3 跑完后去哪里看

推理完成后，结果通常保存在：

```text id="3mlx9f"
/root/autodl-tmp/outputs/GPT4o/<时间戳目录>/
```

例如：

```text id="bcwm0j"
/root/autodl-tmp/outputs/GPT4o/T20260326_G161d400d/
```

你应重点寻找：

```text id="q7595v"
GPT4o_public_subset_extended.xlsx
```

---

### 5.1.4 跑完后先检查什么

执行：

```bash id="3r806k"
ls -lh /root/autodl-tmp/outputs/GPT4o
```

然后查看最新目录中是否生成：

* `.xlsx`
* `.pkl`（有时也会有）

## 6. 对 800 条 caption 实验结果做统一评估

推理结束后，进入统一评估阶段。

---

## 6.1 跑 `extended GT` 的统一评估

执行：

```bash id="a93ya3"
python /root/experiment_suite/evaluate_suite.py \
  --pred-file /root/autodl-tmp/outputs/GPT4o/<时间戳目录>/GPT4o_public_subset_extended.xlsx \
  --out-dir /root/experiment_suite/outputs/eval_800_extended \
  --gt-field answer
```

---

### 6.1.1 参数说明

#### `--pred-file`

指向批量推理产生的原始预测文件。

#### `--out-dir`

指定评估结果输出目录。

#### `--gt-field answer`

表示当前以 `answer` 列为标准答案。
对于 `public_subset_extended.tsv` 来说，`answer` 一般对应 `extended_gt_objects`。

---

## 6.2 跑 `core GT` 的统一评估

如果你想在同一批模型输出上用 stricter GT 再评一次，执行：

```bash id="kbp2h3"
python /root/experiment_suite/evaluate_suite.py \
  --pred-file /root/autodl-tmp/outputs/GPT4o/<时间戳目录>/GPT4o_public_subset_extended.xlsx \
  --out-dir /root/experiment_suite/outputs/eval_800_core \
  --gt-field core_gt_objects
```

---

### 6.2.1 这一步的意义

这一步不会重新跑 GPT-4o，
而是对同一份 `prediction` 用更严格的 GT 重新打分。

这样你就能直接比较：

* `extended GT` 口径下的幻觉率
* `core GT` 口径下的幻觉率

这对判断“高幻觉率有多少来自 GT 不完整”非常关键。

## 8. 跑 800 条 POPE-style 实验

这一部分对应：

* `run_pope_style_800.sh`
* `evaluate_pope_style.py`

目标是从 yes/no probing 角度检测 object hallucination。

---

## 8.1 运行 POPE-style 推理脚本

执行：

```bash id="rq9w10"
bash /root/experiment_suite/run_pope_style_800.sh
```

---

### 8.1.1 这一步做了什么

该脚本会：

1. 从 `public_subset_extended.tsv` 构造 probing 数据集
2. 生成：

```text id="zhznrn"
/root/autodl-tmp/LMUData/public_subset_pope_style.tsv
```

3. 对每张图构造一批 yes/no 问题，例如：

```text id="3togd5"
Is there a truck in the image? Answer only yes or no.
```

4. 调用 VLMEvalKit + GPT-4o 对这些 probing 问题作答

---

### 8.1.2 输出文件位置

运行后，预测结果通常会出现在：

```text id="wsptp3"
/root/autodl-tmp/outputs/GPT4o/<时间戳目录>/
```

例如：

```text id="ud9r6z"
GPT4o_public_subset_pope_style.xlsx
```

---

### 8.1.3 如何判断是否还在运行

如果终端没有回到 shell 提示符，说明通常还在运行。

你也可以另开一个终端检查：

```bash id="wb9jr1"
ps -ef | grep "public_subset_pope_style" | grep -v grep
```

或查看输出目录是否持续增长：

```bash id="zgz8ni"
watch -n 5 'ls -lh /root/autodl-tmp/outputs/GPT4o'
```


重点关注：

* `accuracy`
* `precision`
* `recall`
* `f1`
* 特别是 `fp`

---

## 9.3 为什么 `fp` 特别重要

在 POPE-style probing 中：

* `fp` = 标准答案是 no，但模型答 yes

这最接近：

> 模型“凭空相信某个对象存在”

因此它是 object hallucination 的强信号。

## 10. 结果查看的推荐顺序

建议你每次实验完成后，按以下顺序查看结果：

### 第 1 步：看总体 summary

先看：

* `hallucination_summary.json`
* `pope_style_summary.json`

回答：

* 总体幻觉高不高？
* probing 结果是否也支持这个趋势？

---

### 第 2 步：看分组统计

重点看：

* `group_by_scene`
* `group_by_source`

回答：

* 哪个场景更容易出现 hallucination？
* COCO 和 VG 是否差异明显？

---

### 第 3 步：看典型失败案例

打开：

* `hallucination_details.xlsx`

重点筛：

* `chairi_like_sample` 高的图片
* `hallucinated_objects` 多的图片
* 某个场景下最典型的错误样本

---

### 第 4 步：对比 core vs extended

如果你已经跑了：

* `eval_800_extended`
* `eval_800_core`

就回答：

* strict GT 和 relaxed GT 差多少？
* 幻觉率高到底有多少是 GT 不完整导致的？

## 12. 常见问题与建议

---

### 12.1 评估脚本运行很慢，能否中断？

可以。
如果你给 `evaluate_suite.py` 加了进度条和 JSONL 实时写入，那么：

* 可以 `Ctrl + C`
* 已经写入的 `jsonl` 可以保留
* 重新运行后可继续新一轮评估

---

### 12.2 为什么 `chairi_like` 很高？

不一定表示 GPT-4o 真的严重幻觉。
常见原因包括：

* GT 太短
* 模型输出比 GT 更丰富
* 同义词未归一化
* 某些背景物体并未纳入 GT

---

### 12.3 POPE-style 是否等于官方 POPE？

不是。
你当前跑的是：

> 在自建图像子集上的 POPE-style probing

它有研究价值，但不能直接写成“官方 POPE benchmark 分数”。

---

### 12.4 HallusionBench 能直接用于当前自建集吗？

不能严格这样做。
HallusionBench 是一个独立 benchmark，不是当前自建 caption 数据上的后处理指标。
